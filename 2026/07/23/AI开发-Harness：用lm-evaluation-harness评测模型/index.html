AI开发-Harness：用 lm-evaluation-harness 评测模型

发表于2026-07-23|更新于2026-07-23|AI开发
|字数总计:2.2k|阅读时长:8分钟|阅读量:

前言

模型评测很容易被“感觉”带偏。今天觉得 A 模型回答更自然，明天又觉得 B 模型代码更稳，如果没有固定数据集和指标，最后很难做决策。Harness 这里我主要指 EleutherAI 的 lm-evaluation-harness，它是一个常用的语言模型评测框架。

它的目标是用统一方式在大量任务上测试生成式语言模型。官方 README 里提到，它支持许多标准学术 benchmark，也支持本地模型、API 模型、自定义 prompt 和评测指标。

它适合解决什么问题

如果你在选模型，比如同一个任务到底用 Qwen、Llama、DeepSeek 还是某个微调模型，单纯看排行榜不一定够。排行榜任务和你的业务任务可能不一致，推理框架、量化方式、prompt 格式也会影响结果。

lm-evaluation-harness 的价值在于把评测流程标准化。你可以固定任务、样本、few-shot 设置和模型参数，让多个模型在同一套规则下比较。

一个最小使用流程

安装后可以先列出任务，再选择一个轻量任务跑通。比如用 Hugging Face 后端评测 hellaswag，命令大致是：lm_eval --model hf --model_args pretrained=xxx --tasks hellaswag --device cuda:0 --batch_size 8。

真正用于项目时，我会把命令写成脚本，并把模型版本、数据集版本、commit hash、推理参数一起记录下来。否则过两周再跑一次，结果变了也不知道是哪一层变了。

自定义任务更有价值

通用 benchmark 能提供基础参考，但业务模型最该看业务数据。比如客服场景可以抽一批真实问答做准确率和拒答率评测；代码场景可以设计单测通过率；RAG 场景可以评估引用命中率和答案忠实度。

自定义任务的关键是把“好答案”的标准写清楚。能自动评估就自动评估，不能自动评估也要设计人工打分表，而不是只让人凭感觉看几条样例。

小结

Harness 的意义是让模型选择从主观体验回到可重复实验。对于独立开发者来说，不一定要追求完整评测平台，但至少应该有一组固定命令和固定样本，能在换模型、换 prompt、换推理框架时快速比较。

参考资料：EleutherAI/lm-evaluation-harness。
